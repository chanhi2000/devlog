<p>When writing protocol-oriented Swift, protocols and classes become fairly similar, but they are never the same.</p>
<p>In their basic form, a protocol describes what an unknown type of object can do. You might say it has two or three properties of various types, plus methods. But that protocol never includes anything <em>inside</em> the methods, or provides actual storage for the properties.</p>
<p>In a more advanced form, you can write extensions on your protocols that provide default implementations of the methods. You still can’t provide storage for properties, however.</p>
<p>In comparison, classes are concrete things. While they might adopt protocols – i.e., say they implement the required properties and methods –&nbsp;they aren’t required to do that. You can create objects from classes, whereas protocols are just type definitions. Try to think of protocols as being abstract definitions, whereas classes and structs are real things you can create.</p>
